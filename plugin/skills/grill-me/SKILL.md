# grill-me — 追问式方案拷打

## 核心原则

对用户的方案/设计/决定进行逐层追问，直至达成清晰共识。一次只问一个问题，等答复后继续。每个问题先给出你的推荐答案；若可通过浏览代码库回答，先查代码再答。

## 追问策略

```
第 1 层 → "是什么"  — 方案具体做什么？边界在哪？不做什么？
第 2 层 → "为什么"  — 为什么选这个方案？对比过哪些替代？放弃的理由？
第 3 层 → "怎么做"  — 实现路径是什么？关键风险点？回滚方案？
第 4 层 → "然后呢"  — 做完后怎么验证？谁来维护？后续演进方向？
```

每层问 2~3 个问题后进入下一层。若用户在某一层无法回答，暂停深入，帮用户补全该层信息后再继续。

## 追问模板

### 设计/架构方案
1. "这个方案解决的最核心问题是什么？用一句话说"
2. "有没有更简单的方案能达到同样效果？为什么没选？"
3. "如果这个方案失败，最大的风险点在哪？"
4. "6 个月后回头看，你觉得这个决策还会是对的吗？"

### skill/rule/command 产出
1. "这个产物的使用频次预估？低频（<1次/周）值得固化吗？"
2. "有没有已有的 skill/rule 可以覆盖这个需求？"
3. "触发条件够明确吗？会不会误触发？"
4. "如果 AI 换了一个模型还能正确理解吗？"

### 工作流/习惯沉淀
1. "这个操作重复了几次？是短期密集还是长期稀疏？"
2. "固化后谁来维护？你确定会记得更新它吗？"
3. "这个习惯是阶段性的还是永久的？阶段性的更适合用 memory"

## 被 grow-dream 调用时

当作为 grow-dream 的第⑧步被调用时，聚焦以下方向：
- 针对 grow-dream 分析出的 **高频模式** 逐一追问是否值得固化
- 对 **memory 候选条目** 用"频次≥2 + 跨会话"标准验收
- 对 **skill/command 候选** 用"使用场景 + 维护成本"标准验收
- 输出精简：每个候选给出"建议采纳/建议放弃/需补充信息"的判定
