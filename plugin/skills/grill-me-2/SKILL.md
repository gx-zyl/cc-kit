# grill-me-2 — 深度追问验收

grill-me 的增强版。增加分层追问策略、场景模板、以及被 grow-dream 调用的验收模式。

## 核心原则

一次只问一个问题，等答复后继续。每个问题先给出推荐答案；若可通过浏览代码库回答，先查代码再答。

## 追问策略

```
第 1 层 → "是什么"  — 具体做什么？边界在哪？不做什么？
第 2 层 → "为什么"  — 为什么选这个方案？对比过哪些替代？放弃的理由？
第 3 层 → "怎么做"  — 实现路径？关键风险点？回滚方案？
第 4 层 → "然后呢"  — 做完怎么验证？谁来维护？后续演进？
```

每层问 2~3 个问题后进入下一层。若用户在某层无法回答，暂停深入，帮用户补全信息后继续。

## 追问模板

### 设计/架构方案
1. "这个方案解决的最核心问题是什么？用一句话说"
2. "有没有更简单的方案能达到同样效果？为什么没选？"
3. "如果这个方案失败，最大的风险点在哪？"
4. "6 个月后回头看，这个决策还会是对的吗？"

### skill/rule/command 候选
1. "这个产物的预估使用频次？低频（<1次/周）值得固化吗？"
2. "有没有已有的 skill/rule 可以覆盖？新建 vs 扩展已有？"
3. "触发条件够明确吗？会不会误触发？换模型还能正确理解吗？"
4. "谁来维护？你会记得在业务变更时同步更新吗？"

### 工作流/习惯沉淀
1. "这个操作重复了几次？短期密集还是长期稀疏？"
2. "固化后维护成本多高？你确定会记得更新？"
3. "阶段性的还是永久的？阶段性的更适合 memory 而非 rule"

## 被 grow-dream 调用时（验收模式）

作为 grow-dream 第⑧步被调用时，聚焦验收而非开放式追问：

- 对 grow-dream 输出的 **每个改进候选** 逐一验收
- **高频模式候选**：频次够不够？跨会话适用？值得固化还是 shelfware？
- **memory 候选**：满足"频次≥2 + 跨会话"门槛？已有条目已覆盖？
- **skill/command 候选**：触发条件明确？维护成本可接受？
- 对每个候选输出判定：**建议采纳** / **建议放弃** / **需补充信息**
- 给出简短理由，不过度展开。验收结论追记到 grow-dream 输出的 `## grill-me 验收` 章节
