# grill-dream — 深度追问验收

grill-me + grow-dream 的交集：用 grill-me 的追问哲学，验收 grow-dream 的改进候选。

## 继承自 grill-me 的哲学

以下三条是 grill-me 的核心哲学，grill-dream 完全继承：

> **① 一次只问一个问题，等待答复后再继续。**
> 不连珠炮式追问，不给压迫感。每个问题等用户消化、回答后，再基于答复提出下一个问题。

> **② 每个问题先给出你自己的推荐答案。**
> 不是空洞的"你觉得呢"，而是"我的判断是 X，你怎么看？"。展示你做了功课，降低用户回答成本。

> **③ 如果可以通过浏览代码库回答，就先查代码再回答。**
> 不问"你的项目用的是什么框架"这种可以通过读 `package.json` 回答的问题。查完代码，带着发现去问。

---

在此哲学基础上，grill-dream 增加了**结构化追问**、**验收判定**、**主动识别**三个能力层。

**主动识别与发起**：观察跨会话中用户的常用交互模式，抽象为提问式链路矩阵。当识别到匹配场景时，不等用户触发，主动发起追问——把"用户每次问 AI"变成"AI 主动问用户"，降低用户反复表达的成本。

## 追问策略：四层递进

```
第 1 层 → "是什么"  — 具体做什么？边界在哪？不做什么？
第 2 层 → "为什么"  — 为什么选这个方案？对比过哪些替代？放弃的理由？
第 3 层 → "怎么做"  — 实现路径？关键风险点？回滚方案？
第 4 层 → "然后呢"  — 做完怎么验证？谁来维护？后续演进方向？
```

每层问 2~3 个问题后进入下一层。若用户在某层无法回答，暂停深入，帮用户补全该层信息后再继续。

**每层仍遵守三条哲学**：一次一问、先给答案、先查代码。

## 追问模板

### 设计/架构方案
1. "这个方案解决的最核心问题是什么？用一句话说" → 我的理解是……
2. "有没有更简单的方案能达到同样效果？为什么没选？" → 我看到的替代有……
3. "如果这个方案失败，最大的风险点在哪？" → 我判断风险点在……
4. "6 个月后回头看，这个决策还会是对的吗？"

### skill/rule/command 候选
1. "这个产物的预估使用频次？低频（<1次/周）值得固化吗？"
2. "有没有已有的 skill/rule 可以覆盖？新建 vs 扩展已有？" → 我扫描了现有，……
3. "触发条件够明确吗？会不会误触发？换模型还能正确理解吗？"
4. "谁来维护？你会在业务变更时记得同步更新吗？"

### 工作流/习惯沉淀
1. "这个操作重复了几次？短期密集还是长期稀疏？" → 我从对话中统计到……
2. "固化后维护成本多高？"
3. "阶段性的还是永久的？阶段性的更适合 memory 而非 rule"

## 提问式链路矩阵

将用户高频交互模式抽象为"场景→触发信号→追问链路"矩阵，主动发起，不等用户问：

| 场景 | 触发信号 | 追问链路 |
|------|---------|---------|
| 新建 skill | 对话中出现"写个 skill""创建 skill" | 频次?→ 已有覆盖?→ 触发明确?→ 维护成本? |
| 修改 rule | 用户反复纠正同一行为 | 纠正了几次?→ 跨项目通用?→ 写入哪层? |
| 沉淀 memory | 跨 ≥2 会话出现相同偏好 | 规律稳定?→ 已有条目覆盖?→ MEMORY.md 更新? |
| 方案决策 | 用户提出"要不要做 X" | 是什么→ 为什么→ 怎么做→ 然后呢（四层递进） |

每识别到一个匹配场景，主动发起第一个问题，不等 `/grill-dream` 或 `grow-dream` 触发。

## 独立使用时

当用户直接触发（非 grow-dream 调用）时，走完整四层递进追问：

1. 先理解用户方案/设计/决定的上下文
2. 从第 1 层开始逐层追问，每层 2-3 问后递进
3. 不限于验收判定——也包含方案拷打、假设挑战、替代探索
4. 结束时给出整体评估和建议

## 被 grow-dream 调用时（验收模式）

作为 grow-dream 第⑧步被调用时，聚焦验收判定：

- 对 grow-dream 输出的 **每个改进候选** 逐一验收
- 仍遵守三条哲学：一次一个候选、先查已有文件是否覆盖、给出判定理由
- 验收维度：
  - **高频模式候选**：频次够不够？跨会话通用还是本次特例？
  - **memory 候选**：满足"频次≥2 + 跨会话"双门槛？已有条目已覆盖？
  - **skill/command 候选**：触发条件明确？维护成本可接受？与其他 skill 重叠？
- 对每个候选输出判定：**建议采纳** / **建议放弃** / **需补充信息**
- 验收结论追加到 grow-dream 输出的 `## grill-dream 验收` 章节
